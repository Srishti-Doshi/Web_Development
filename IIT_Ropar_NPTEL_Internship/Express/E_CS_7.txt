Title: Request Validation in Express
Case Study: BrightFuture University Admissions


------------------------------------------------------------
1. Problem Statement: Chaos at BrightFuture University

BrightFuture University receives thousands of online applications every year.

As the number of applicants increased, so did the number of problems:

• Some students submitted forms with missing grades.
• Some entered incorrect birthdates like “202”.
• Some uploaded wrong files instead of transcripts.
• Some applications were processed even though contact information was incomplete.
• Sometimes the system rejected students incorrectly because it failed to validate properly.

The biggest issue was fairness.

A student might lose admission simply because the system didn’t properly check their form. On the other hand, incomplete applications were sometimes processed, wasting time and resources.

The university needed a system that:

• Checks every application before processing.
• Ensures all required fields are present.
• Gives clear error messages to students.
• Prevents invalid or incomplete data from entering the system.

3. What Is Request Validation? (Simple Explanation)

Request validation means checking user input before using it.

When a client sends data to the server:

Request → Validation Middleware → Business Logic → Response

If the data is invalid, we immediately stop the request and send an error.

This prevents:

• Incomplete forms
• Invalid email formats
• Wrong data types
• Malicious input
• System crashes due to bad data

It follows a simple principle:

"Validate first, process later."

4. Designing the Application Structure

First, we define what a valid application should look like.

interface Application {
  name: string;
  email: string;
  birthdate: string;
  grades: number[];
  essay: string;
  recommendationLetter: string; // URL
}

Each field has a purpose:

• name → Required for identification
• email → Needed for communication
• birthdate → Must be a real date
• grades → At least one subject
• essay → Must meet minimum length
• recommendationLetter → Must be a valid link

If any of these are missing or incorrect, the application should not proceed.

5. Implementing Validation Using express-validator

First, install the library:

npm install express-validator


------------------------------------------------------------
A. Defining Validation Rules

import { body, validationResult } from "express-validator";

const applicationValidation = [

  body("name")
    .isString()
    .notEmpty()
    .withMessage("Name is required"),

  body("email")
    .isEmail()
    .withMessage("Valid email is required"),

  body("birthdate")
    .isISO8601()
    .withMessage("Birthdate must be in YYYY-MM-DD format"),

  body("grades")
    .isArray({ min: 1 })
    .withMessage("At least one grade is required"),

  body("grades.*")
    .isNumeric()
    .withMessage("All grades must be numbers"),

  body("essay")
    .isLength({ min: 100 })
    .withMessage("Essay must be at least 100 characters long"),

  body("recommendationLetter")
    .isURL()
    .withMessage("A valid recommendation letter link is required")
];

Explanation:

Each rule checks one requirement.

If any rule fails, the system stops the request immediately and returns a clear error message.


------------------------------------------------------------
B. Handling Validation in Route

app.post("/apply", applicationValidation, (req, res) => {

  const errors = validationResult(req);

  if (!errors.isEmpty()) {
    return res.status(400).json({
      status: "error",
      errors: errors.array()
    });
  }

  // If validation passes
  res.status(200).json({
    status: "success",
    message: "Application received successfully"
  });
});

Explanation:

• validationResult(req) collects all errors.
• If errors exist → return 400 Bad Request.
• If no errors → continue processing.

This ensures only valid applications move forward.

6. Example Error Response

If a student submits invalid data:

{
  "status": "error",
  "errors": [
    {
      "msg": "Valid email is required",
      "param": "email"
    },
    {
      "msg": "Essay must be at least 100 characters long",
      "param": "essay"
    }
  ]
}

This gives students clear instructions on what to fix.

No confusion.
No guesswork.

9. Common Pitfalls & Best Practices

Mistake:
Validating only some fields.

Best Practice:
Validate every required field.

Mistake:
Returning generic error messages.

Best Practice:
Provide specific and actionable feedback.

Mistake:
Running business logic before validation.

Best Practice:
Always validate first.

Mistake:
Ignoring array validation.

Best Practice:
Use grades.* to validate nested values.


------------------------------------------------------------
10. Key Takeaways

• Request validation ensures data quality.
• Always validate before processing.
• Use express-validator middleware for clean structure.
• Return consistent error responses.
• Good validation improves user experience and fairness.
